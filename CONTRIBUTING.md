# Руководство по внесению изменений

Спасибо за интерес к проекту Flash Cards! Это руководство поможет вам внести свой вклад в развитие приложения.

## Как начать

1. Fork репозитория
2. Клонируйте свой fork
3. Создайте новую ветку для ваших изменений
4. Внесите изменения
5. Протестируйте изменения
6. Создайте Pull Request

## Настройка окружения

```bash
# Клонирование
git clone <your-fork-url>
cd flash-cards

# Запуск через Docker
docker compose up --build

# Или локально
npm install
docker compose up postgres
npx prisma migrate dev
npm run dev
```

## Структура веток

- `main` - production-ready код
- `develop` - development ветка
- `feature/*` - новый функционал
- `bugfix/*` - исправление багов
- `hotfix/*` - срочные исправления

## Соглашения о коде

### TypeScript

- Строгая типизация
- Явные типы для всех функций
- Избегайте `any`
- Используйте интерфейсы для сложных типов

### React

- Функциональные компоненты
- Hooks вместо классов
- Декомпозиция больших компонентов
- Props типизация

### Стиль кода

- 2 пробела для отступов
- Точка с запятой в конце строк
- Single quotes для строк
- Trailing commas

### Именование

- Компоненты: PascalCase
- Функции: camelCase
- Константы: UPPER_SNAKE_CASE
- Файлы компонентов: kebab-case.tsx
- Файлы утилит: kebab-case.ts

## Git commit сообщения

Формат: `<type>(<scope>): <subject>`

### Типы

- `feat`: Новый функционал
- `fix`: Исправление бага
- `docs`: Изменения в документации
- `style`: Форматирование, отсутствующие точки с запятой и т.д.
- `refactor`: Рефакторинг кода
- `test`: Добавление тестов
- `chore`: Обновление зависимостей, сборка и т.д.

### Примеры

```
feat(training): add stage 5 with voice recognition
fix(words): correct translation caching issue
docs(readme): update installation instructions
refactor(api): simplify error handling
```

## Тестирование

Перед созданием PR убедитесь, что:

- [ ] Код компилируется без ошибок
- [ ] Приложение запускается через Docker
- [ ] Новый функционал работает корректно
- [ ] Не сломан существующий функционал
- [ ] Код соответствует style guide

## Pull Request процесс

1. **Создание PR**
    - Опишите изменения
    - Укажите связанные issues
    - Добавьте скриншоты (если применимо)

2. **Шаблон PR**

```markdown
## Описание

Краткое описание изменений

## Тип изменений

- [ ] Новый функционал
- [ ] Исправление бага
- [ ] Документация
- [ ] Рефакторинг

## Тестирование

Описание того, как были протестированы изменения

## Checklist

- [ ] Код следует style guide проекта
- [ ] Изменения протестированы
- [ ] Документация обновлена
- [ ] Commit сообщения следуют конвенции
```

3. **Code Review**
    - Ожидайте review от maintainer
    - Внесите запрошенные изменения
    - Получите approval

4. **Merge**
    - После approval изменения будут смержены в основную ветку

## Добавление новых зависимостей

При добавлении новых npm пакетов:

1. Объясните необходимость в PR
2. Проверьте размер bundle
3. Убедитесь в совместимости с Next.js
4. Обновите `package.json` и документацию

## API изменения

При изменении API endpoints:

1. Обновите документацию в USAGE.md
2. Сохраните обратную совместимость или версионируйте API
3. Добавьте миграции БД при необходимости

## Database миграции

```bash
# Создание миграции
npx prisma migrate dev --name your_migration_name

# Проверка миграции
npx prisma migrate status

# Применение в production
npx prisma migrate deploy
```

## Документация

При добавлении функционала обновите:

- README.md - если изменился запуск/установка
- USAGE.md - если изменился пользовательский интерфейс
- ARCHITECTURE.md - если изменилась архитектура
- Код комментарии - для сложной логики

## Вопросы и помощь

Если у вас есть вопросы:

1. Проверьте существующую документацию
2. Поищите в Issues
3. Создайте новый Issue с тегом `question`

## Код поведения

- Будьте вежливы и уважительны
- Конструктивная критика приветствуется
- Помогайте другим участникам
- Фокусируйтесь на коде, а не на личностях

## Лицензия

Внося изменения, вы соглашаетесь с лицензией проекта.

Спасибо за ваш вклад! 🎉
